## תשובת כלי ה-AI – הסבר על בדיקות יחידה

### מהן בדיקות יחידה (Unit Tests)?

בדיקות יחידה הן בדיקות קטנות וממוקדות שמטרתן לבדוק חלק ספציפי וקטן מתוך התוכנה, בדרך כלל פונקציה או מתודה אחת בתוך מחלקה.

המטרה היא לוודא שכל פונקציה עובדת נכון בפני עצמה, בלי תלות בשאר חלקי המערכת.

בדיקת יחידה טובה צריכה להיות:

- **בודדת** – בודקת רק את הלוגיקה של הפונקציה הספציפית.
- **אוטומטית** – ניתן להריץ אותה שוב ושוב בעזרת כלי כמו Gradle.
- **חוזרת על עצמה** – בכל הרצה, עבור אותו קלט, מתקבלת אותה תוצאה.
- **ברורה** – קל להבין מה היא בודקת ומה התוצאה המצופה.

---

## ספריית JUnit

ב-Java נהוג להשתמש בספריית **JUnit** כדי לכתוב בדיקות יחידה.

JUnit מספקת:

- סימון של מתודות בדיקה בעזרת `@Test`.
- פונקציות בדיקה מסוג Assertions, למשל:
  - `assertEquals`
  - `assertTrue`
  - `assertFalse`
  - `assertNull`
  - `assertNotNull`

פונקציות אלו משוות בין התוצאה שהתקבלה בפועל לבין התוצאה שאנחנו מצפים לקבל.

---

## מבנה בסיסי של בדיקת יחידה

בדיקת יחידה בדרך כלל מחולקת לשלושה שלבים:

1. **Arrange** – הכנת האובייקטים והנתונים הדרושים לבדיקה.
2. **Act** – הפעלת הפונקציה שרוצים לבדוק.
3. **Assert** – בדיקה שהתוצאה שהתקבלה היא התוצאה הרצויה.

---

## דוגמה למחלקה שנבדקת

בדוגמה זו קיימת מחלקה בשם `Thief`, ובה פונקציה בשם `canSteal`. הפונקציה מקבלת ערך של חפץ ובודקת האם האזעקה פעילה.

```java
public class Thief {
    public boolean canSteal(int itemValue, boolean isAlarmActive) {
        if (isAlarmActive) {
            return false; // אי אפשר לגנוב אם האזעקה פעילה
        }

        if (itemValue > 100) {
            return true; // שווה לגנוב רק אם הערך מעל 100
        }

        return false; // אחרת, לא שווה לגנוב
    }
}
```

---

## דוגמה למחלקת בדיקות ב-JUnit

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class ThiefTest {

    @Test
    public void testCanSteal_AlarmIsActive_ReturnsFalse() {
        // 1. Arrange - הכנות
        Thief thief = new Thief();

        // 2. Act - הפעלת הפונקציה הנבדקת
        boolean result = thief.canSteal(150, true);

        // 3. Assert - אימות התוצאה
        assertFalse(result, "כאשר האזעקה פעילה הפונקציה אמורה להחזיר false");
    }
}
```

### הסבר הדוגמה

- יצרנו אובייקט מסוג `Thief`.
- הפעלנו את הפונקציה `canSteal` עם ערך חפץ `150` וכאשר האזעקה פעילה.
- מכיוון שהאזעקה פעילה, מצופה שהתוצאה תהיה `false`.
- השתמשנו ב-`assertFalse` כדי לוודא שהתוצאה אכן שקרית.

## כיצד יודעים אם הבדיקות מספקות?

כדי לדעת שהבדיקות באמת מכסות את מה שצריך, מסתמכים על כמה עקרונות מרכזיים:

### 1. כיסוי קוד (Code Coverage)

כיסוי קוד הוא מדד כמותי, שמוצג בדרך כלל באחוזים, ומציין כמה משורות הקוד הופעלו במהלך הרצת הבדיקות.

לדוגמה, אם פונקציה מכילה 10 שורות קוד והבדיקות הפעילו 8 מתוכן, הכיסוי הוא 80%.

השאיפה היא להגיע לכיסוי גבוה ככל האפשר, ובמיוחד לוודא שכל התנאים החשובים נבדקים.

### 2. בדיקת מקרי קצה (Edge Cases)

בדיקות מספקות לא בודקות רק את המצב הרגיל שבו הכל תקין, אלא גם מקרי קצה.

לדוגמה, עבור הפונקציה `canSteal` נרצה לבדוק:

- מה קורה כאשר `itemValue` גדול מ-100.
- מה קורה כאשר `itemValue` שווה בדיוק ל-100.
- מה קורה כאשר `itemValue` קטן מ-100.
- מה קורה כאשר `isAlarmActive` הוא `true`.
- מה קורה כאשר `isAlarmActive` הוא `false`.

### 3. עצמאות הבדיקות

כל בדיקה צריכה להיות עצמאית ולא תלויה בבדיקות אחרות.

כלומר, כל בדיקה מכינה לעצמה את הנתונים, מריצה את הפונקציה, ובודקת את התוצאה שלה.

---

## איך בודקים שכל נתיבי הפונקציה נבדקו?

כיסוי נתיבים (Branch/Path Coverage) הוא שלב מתקדם יותר מכיסוי שורות. המטרה היא לוודא שכל תנאי לוגי בפונקציה נבדק גם במצב `true` וגם במצב `false`.

בפונקציה `canSteal` קיימים כמה נתיבים אפשריים:

1. **נתיב 1:** `isAlarmActive == true`  
   במקרה זה הפונקציה מחזירה `false` מיד.

2. **נתיב 2:** `isAlarmActive == false` וגם `itemValue > 100`  
   במקרה זה הפונקציה מחזירה `true`.

3. **נתיב 3:** `isAlarmActive == false` וגם `itemValue <= 100`  
   במקרה זה הפונקציה מחזירה `false`.

כדי לוודא שכל הנתיבים נבדקו, יש לכתוב בדיקה נפרדת עבור כל אחד מהם.

---

## דוגמאות לבדיקות עבור כל הנתיבים

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class ThiefTest {

    @Test
    public void testCanSteal_AlarmIsActive_ReturnsFalse() {
        Thief thief = new Thief();
        boolean result = thief.canSteal(150, true);
        assertFalse(result, "כאשר האזעקה פעילה הפונקציה אמורה להחזיר false");
    }

    @Test
    public void testCanSteal_AlarmInactiveAndItemValueAbove100_ReturnsTrue() {
        Thief thief = new Thief();
        boolean result = thief.canSteal(150, false);
        assertTrue(result, "כאשר האזעקה לא פעילה וערך החפץ מעל 100 הפונקציה אמורה להחזיר true");
    }

    @Test
    public void testCanSteal_AlarmInactiveAndItemValueEquals100_ReturnsFalse() {
        Thief thief = new Thief();
        boolean result = thief.canSteal(100, false);
        assertFalse(result, "כאשר ערך החפץ שווה ל-100 הפונקציה אמורה להחזיר false");
    }

    @Test
    public void testCanSteal_AlarmInactiveAndItemValueBelow100_ReturnsFalse() {
        Thief thief = new Thief();
        boolean result = thief.canSteal(50, false);
        assertFalse(result, "כאשר ערך החפץ קטן מ-100 הפונקציה אמורה להחזיר false");
    }
}
```

## שימוש בכלי Coverage

בסביבות פיתוח כמו IntelliJ או Eclipse ניתן להריץ את הבדיקות עם כלי Coverage.

לאחר ההרצה, הכלי מציג בצבעים אילו שורות קוד נבדקו:

- ירוק – שורות שהורצו במהלך הבדיקות.
- אדום – שורות שלא נבדקו כלל.
- צהוב – תנאי שנבדק רק בחלק מהאפשרויות שלו.

כך ניתן לראות האם קיימים נתיבים שלא כוסו ולכתוב עבורם בדיקות נוספות.
